D1: Agentic Architecture & Orchestration

The highest-weighted domain — understand how agents perceive, reason, and act; multi-agent coordination patterns; and hooks for deterministic control.

⚡ Claude Code in Action 🔧 Building with Claude API
Core Topics
1Agentic Control Flow
  • The agentic loop: perception → reasoning → action, repeated until task completion or stop condition
  • Tool calls are the primary mechanism for taking actions in the environment (file edits, bash commands, web searches)
  • Agents decide when to call tools, which tool to call, and how to use results
  • Stop conditions: task complete, max iterations reached, confidence below threshold, human review required
  • Understand computer use vs. standard structured tools — computer use is for GUI interaction, standard tools for APIs/files
agentic loop tool_use block tool_result stop_reason
2Multi-Agent Orchestration
  • Coordinator agent: receives the high-level task, breaks it into subtasks, delegates to subagents, synthesizes results
  • Subagent: receives a scoped subtask, executes with its own tool set, returns a structured result
  • Patterns: fan-out (parallel subagents), sequential chain, hierarchical tree
  • Fan-out enables parallelism but requires the coordinator to merge potentially conflicting results
  • Subagents should have minimal required permissions — least-privilege principle
  • Claude SDK: anthropic.beta.messages.batches for async multi-agent workloads; direct API calls for orchestration
coordinator subagent fan-out Agent SDK
3Hooks for Deterministic Enforcement
  • PreToolUse: runs before any tool call; can block the call or modify parameters; return {"decision":"block","reason":"..."} to deny
  • PostToolUse: runs after a tool returns; can inspect results; inject additional context into the next turn
  • Stop: fires when the model signals it wants to stop; use to persist state or trigger cleanup
  • SubagentStop: fires when a subagent finishes; allows coordinator to collect and process results
  • Hooks receive structured JSON via stdin; must return JSON on stdout
  • Hook scripts run as shell processes — keep them fast (<200ms) to avoid latency
  • Use hooks for: audit logging, cost guardrails, safety filters, enforcing code review gates
PreToolUse PostToolUse Stop SubagentStop stdin/stdout JSON
4Session & Memory Management
  • Sessions are stateless by default — each API call is independent unless you pass prior messages
  • In-context memory: include conversation history in the messages array (limited by context window)
  • External memory: write summaries to files or databases; load relevant facts at session start
  • Session files: write task state to disk at each turn so progress survives restarts
  • Use the system prompt to inject persistent facts, user preferences, and project context
  • Rolling summary: compress old turns into a summary block to stay within context limits
messages[] system prompt session files rolling summary

D2: Tool Design & MCP Integration

Writing effective tool definitions, structured error handling, tool_choice control, and configuring MCP servers in Claude Code.

🚀 MCP Advanced Topics 🔌 Introduction to MCP
Core Topics
1Writing Effective Tool Descriptions
  • The description field is the most important part of a tool definition — Claude reads it to decide when to call the tool
  • Be specific: state the purpose, what inputs are required, what the tool returns, and when not to use it
  • Include examples of ideal inputs in the description when the format is non-obvious
  • Match description length to tool complexity — simple tools need short descriptions; complex tools need more detail
  • Parameter descriptions should specify valid ranges, formats, and constraints
  • Avoid vague descriptions like "does X stuff" — be precise about behavior and side effects
name description input_schema
2Structured Error Responses
  • Return errors as structured tool results, not Python exceptions — use isError: true in the tool_result block
  • Include an error code (machine-readable) and a human-readable message in every error response
  • Provide actionable guidance — tell the model what to do next, not just what went wrong
  • Distinguish error types: validation errors, auth errors, rate limits, transient failures
  • For transient errors, include a retry_after hint so the model knows when to retry
isError: true tool_result error_code
3Tool Distribution & tool_choice
  • tool_choice: "auto" — default; model decides whether to use a tool
  • tool_choice: "any" — model must use at least one tool (prevents pure-text responses)
  • tool_choice: {"type":"tool","name":"..."} — forces a specific tool call (useful for structured output)
  • Limit the number of tools exposed to what the model actually needs — too many tools increase cost and confusion
  • Group related tools; remove or gate tools not relevant to the current task
tool_choice "auto" "any" "tool"
4MCP Server Configuration
  • MCP servers are configured in claude_desktop_config.json (desktop) or .claude/settings.json (project)
  • Transport types: stdio (subprocess, recommended for local servers) vs. SSE (HTTP, for remote servers)
  • Each server defines its own tools — Claude sees them alongside built-in tools
  • Servers can expose resources (read-only data) and prompts (reusable templates) in addition to tools
  • Auth: pass credentials via environment variables in the server config; never hardcode in config files
claude_desktop_config.json stdio SSE resources
5Built-in Claude Code Tools
  • Read: reads a file by absolute path; returns line-numbered content
  • Write: creates or overwrites a file; always read first before overwriting
  • Edit / MultiEdit: performs exact string replacements; safer than Write for partial changes
  • Bash: runs shell commands; use for git, npm, grep, and other CLI operations
  • Glob: finds files matching a pattern; returns matching paths
  • Grep: searches file contents for a pattern; returns matching lines with context
  • Know which tool to prefer: Read > cat, Edit > sed, Grep > manual search
Read Write Edit Bash Glob Grep

D3: Claude Code Configuration & Workflows

Mastering CLAUDE.md hierarchy, custom commands, plan mode, permission management, and CI/CD integration.

⚡ Claude Code in Action 💻 Claude Code 101
Core Topics
1CLAUDE.md Hierarchy
  • Global: ~/.claude/CLAUDE.md — applies to all projects on the machine
  • Project root: .claude/CLAUDE.md or CLAUDE.md in the repo root — applies to the whole project
  • Directory-level: CLAUDE.md in any subdirectory — applies only when working in that subtree
  • More specific (deeper) files take precedence and can override or extend parent instructions
  • Use CLAUDE.md for: coding standards, preferred libraries, forbidden patterns, project context, tool restrictions
  • Instructions in CLAUDE.md are injected into the system prompt at session start
~/.claude/CLAUDE.md .claude/CLAUDE.md directory CLAUDE.md
2Path-Specific Glob Rules
  • Use glob-pattern section headers in CLAUDE.md to scope instructions to specific files
  • Example: [*.test.ts] section applies only when editing TypeScript test files
  • Useful for: test-file conventions, migration file rules, auto-generated file warnings
  • Glob patterns follow standard shell glob syntax; ** matches any depth
  • Multiple sections can match — all matching sections apply (most specific wins on conflicts)
[*.test.ts] glob patterns **
3Custom Slash Commands & Skills
  • Create .claude/commands/ directory; each .md file becomes a /commandname slash command
  • Commands are Markdown files — they can contain instructions, templates, checklists
  • Use $ARGUMENTS placeholder to accept user-provided arguments in commands
  • Skills are more powerful commands that can invoke the full Claude Code tool loop
  • Commands are local to the project; share via version control for team consistency
  • Examples: /review for PR review checklist, /test to run test suites, /deploy to run a deploy sequence
.claude/commands/ $ARGUMENTS /commandname
4Plan Mode vs Direct Execution
  • Plan mode: Claude presents a step-by-step plan before executing any actions; user can approve, edit, or reject
  • Toggle plan mode: Shift+Tab in the REPL, or use /plan slash command
  • Use plan mode for: complex multi-step tasks, destructive operations, unclear requirements
  • Direct execution: Claude acts immediately without a planning step — faster for simple tasks
  • In plan mode, Claude cannot make tool calls until the plan is approved — a safety guardrail
  • The --dangerously-skip-permissions flag bypasses all permission prompts — only for CI/CD automation
plan mode Shift+Tab /plan --print
5Claude Code in CI/CD Pipelines
  • Use non-interactive mode: claude --print "task description" — outputs result to stdout and exits
  • Set ANTHROPIC_API_KEY environment variable; never hardcode credentials
  • Use --allowedTools to restrict which tools CI can invoke — principle of least privilege
  • Use --dangerously-skip-permissions in fully automated CI pipelines where no human can approve prompts
  • Pipe task descriptions from files: claude --print "$(cat task.txt)"
  • Exit codes: 0 = success, non-zero = error; parse stdout for structured results
--print ANTHROPIC_API_KEY --allowedTools

D4: Prompt Engineering & Structured Output

Forcing structured output via tools, few-shot examples, validation-retry loops, and the Message Batches API.

🔧 Building with the Claude API ☁️ Bedrock (prompt eng sections)
Core Topics
1Structured Output via tool_use
  • Force structured JSON output by defining a tool with a JSON schema and setting tool_choice to that tool
  • The model must respond with a valid tool_use block matching the schema — not a free-text response
  • JSON schema supports: required fields, enum values, nested objects, arrays, descriptions per field
  • Use additionalProperties: false to prevent the model from adding extra fields
  • Validate the output against your schema programmatically and retry with feedback if it fails
  • Alternative: ask for JSON in the prompt with "Respond with valid JSON only" — less reliable but simpler
tool_use input_schema tool_choice additionalProperties
2Few-Shot Prompting
  • Provide input/output example pairs in the prompt to demonstrate desired format and style
  • Include diverse examples covering edge cases — don't just show the happy path
  • Format consistently: use a clear delimiter between examples (e.g., --- or XML-like tags)
  • Few-shot examples work best when the task has a clear pattern that can be demonstrated
  • For complex tasks, 3–5 examples often saturate gains; more rarely helps significantly
  • Interleave examples in user/assistant message turns for conversational tasks
few-shot in-context examples input/output pairs
3Explicit Criteria & Rubrics
  • Define clear acceptance criteria in the system prompt — what makes a good output vs. a bad one
  • Use checklists and rubrics so the model can self-evaluate before responding
  • Specify format, length, tone, level of detail, and structure explicitly
  • For classification tasks, enumerate all valid labels and their definitions
  • Ask the model to explain its reasoning (chain-of-thought) for tasks where traceability matters
acceptance criteria rubric chain-of-thought
4Validation-Retry Loops
  • Check model output against programmatic criteria (schema validation, regex, business rules)
  • If validation fails, send a new message with specific feedback: what was wrong and what to fix
  • Include the failed output in the feedback so the model can see exactly what it produced
  • Cap retries at 3–5 attempts to avoid infinite loops; log failures for monitoring
  • For structured output: re-inject the JSON schema in the retry message as a reminder
  • Track retry counts — high retry rates signal prompt quality issues to investigate
retry loop validation feedback max_retries
5Message Batches API
  • Process up to 10,000 requests asynchronously in a single batch — ideal for bulk operations
  • Batches are 50% cheaper than real-time API calls — use for non-time-sensitive workloads
  • Submit a batch, get a batch_id, then poll /v1/messages/batches/{id} for completion
  • Each request in the batch can have different models, prompts, and parameters
  • Results are returned as a JSONL file once the batch completes (can take minutes to hours)
  • Use for: dataset labeling, bulk content generation, offline evaluation pipelines
batches API batch_id JSONL results

D5: Context Management & Reliability

Preserving critical information across long sessions, escalation decisions, error propagation in multi-agent systems, and human review patterns.

🔧 Building with the Claude API ⚡ Claude Code in Action
Core Topics
1Preserving Critical Info Across Long Sessions
  • Claude's context window is finite — long sessions degrade if important info scrolls out of range
  • Explicitly summarize key decisions at the end of each major task phase: "Summary: we decided X because Y"
  • Load critical facts into the system prompt — it's always in context and never truncated
  • Use external memory (files, vector databases) for facts that must survive across separate sessions
  • Rolling window: when approaching context limits, compress old turns into a structured summary and truncate the raw history
  • Prioritize: task state > constraints > history when deciding what to keep in context
context window system prompt rolling summary external memory
2Escalation Decision-Making
  • Define escalation criteria explicitly in the system prompt — don't rely on the model to infer them
  • Always escalate on: ambiguous destructive actions, conflicting instructions, missing required context
  • Use confidence thresholds — if the model's confidence in its interpretation is below a threshold, pause and ask
  • Implement interrupt hooks (Stop hook) to create human review checkpoints in automated pipelines
  • Log escalation events with full context so humans can quickly understand why the model stopped
escalation criteria confidence threshold Stop hook human-in-the-loop
3Error Propagation in Multi-Agent Systems
  • Errors in subagents must propagate cleanly to the coordinator — don't silently swallow failures
  • Use structured error types (not plain strings) so coordinators can handle them programmatically
  • Coordinator should decide: retry the subagent, use a fallback, or escalate to human
  • Partial failures in fan-out: decide whether to proceed with partial results or abort the whole task
  • Idempotent tool calls — design tools so retrying them is safe; avoid side effects on failure
error propagation structured error idempotent fallback
4Human Review & Confidence Calibration
  • Build explicit review checkpoints into long agentic workflows — don't run end-to-end without oversight
  • Ask the model to output a confidence score with its answer; flag low-confidence outputs for review
  • Use PostToolUse hooks to inspect tool results and inject human approval requests when needed
  • For high-stakes actions (deleting data, sending emails, making purchases), require explicit human confirmation
  • Track error rates and retry rates as proxy metrics for reliability — spikes indicate prompt or tool issues
review checkpoint confidence score PostToolUse human confirmation
