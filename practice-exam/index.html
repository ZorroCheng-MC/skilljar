Scenario 1 — Customer Support Resolution Agent
Q1 D1 D2 Task 1.4 + 2.3
Production data shows 12% of cases skip get_customer and call lookup_order directly, causing misidentified accounts. What is the most effective fix?
Why A: Programmatic enforcement provides deterministic guarantees — the tool simply cannot be called out of order. Prompt-based approaches (B, C) are probabilistic and will still fail some percentage of the time. A routing classifier (D) doesn't fix the underlying sequencing problem.
Reference Topics
Agentic control flow & tool ordering → Claude Code in Action — Hooks chapter
Tool design & prerequisites → Building with the Claude API — Tool schemas, Fine grained tool calling
Q2 D2 Task 2.1
An agent calls get_customer for order queries instead of lookup_order. Both tools have minimal descriptions. What is the most important first step?
Why B: Tool descriptions are the primary mechanism by which Claude selects tools — fix the root cause first. Few-shot examples (A) add weight but don't address the missing specification. Adding infrastructure (C, D) before clarifying descriptions is premature.
Reference Topics
Writing effective tool descriptions → Building with the Claude API — Tool schemas, Introducing tool use
MCP tool definitions → Introduction to MCP — Defining tools with MCP
Q3 D5 Task 5.2
The agent achieves 55% first-contact resolution (target: 80%). It escalates simple cases and handles complex ones itself — miscalibrated escalation. What is the best fix?
Why A: Explicit criteria with examples directly addresses the unclear decision boundaries. Self-reported confidence (B) is known to be poorly calibrated in LLMs. A separate classifier (C) adds complexity and latency. Sentiment (D) is a proxy signal, not the root cause.
Reference Topics
Escalation design & human-in-the-loop → Building with the Claude API — Workflows vs agents, Agents and tools
Prompt engineering for decision making → Building with the Claude API — Being clear and direct, Providing examples
Scenario 2 — Code Generation with Claude Code
Q4 D3 Task 3.2
You want to create a /review slash command that is automatically available to every developer when they clone the repository. Where should you put it?
Why A: Project-scoped commands in .claude/commands/ are version-controlled and automatically available to every developer who clones the repo. User-scoped commands (B) require each dev to set them up individually. CLAUDE.md (C) is for context/instructions, not slash commands.
Reference Topics
Custom slash commands (project vs user scope) → Claude Code in Action — Custom commands
Skills and commands → Claude Code 101 — Skills
Q5 D3 Task 3.4
You are restructuring a monolithic application into microservices — dozens of files need changes and major architectural decisions must be made. What is the best approach?
Why A: Plan mode is designed for complex, multi-file tasks with significant architectural decisions — review the approach before any changes are made. Incremental direct execution (B, D) risks compounding mistakes across dozens of files.
Reference Topics
Plan mode (explore → plan → code → commit) → Claude Code 101 — The explore → plan → code → commit workflow
Agentic workflows & multi-file changes → Claude Code in Action — Making changes
Q6 D3 Task 3.3
Your codebase has different conventions for React components, API handlers, database models, and test files across the project. How should you configure Claude Code to apply the correct conventions based on file type?
Why A: .claude/rules/ with glob frontmatter (e.g., globs: ["**/*.test.tsx"]) applies conventions based on file path patterns regardless of directory. A single CLAUDE.md (B) applies globally. Subdirectory CLAUDE.md files (D) miss files in other directories.
Reference Topics
Rule files with glob patterns → Claude Code in Action — Adding context
CLAUDE.md and context files → Claude Code 101 — The CLAUDE.md file
Scenario 3 — Multi-Agent Research System
Q7 D1 Task 1.2 + 1.6
A multi-agent research system covering "creative industries" produces results focused only on visual arts, despite all subagents functioning correctly. Logs show the coordinator decomposed the topic into only visual art subtasks. What is the root cause?
Why B: The logs show the coordinator only decomposed into visual arts subtasks — the subagents worked correctly but were given the wrong assignments. The synthesis agent (A) can only synthesize what it received. Search (C) and document agents (D) can only process what they are asked to find.
Reference Topics
Multi-agent orchestration & task decomposition → Building with the Claude API — Agents and workflows, Parallelization workflows
Orchestrator vs subagent roles → Claude Code in Action — The Claude Code SDK
Q8 D5 Task 5.3
A web search subagent times out. How should you design error propagation so the coordinator can make intelligent recovery decisions?
Why A: Structured error context enables the coordinator to make intelligent recovery decisions — retry with a different query, use an alternative source, or adjust scope. Generic errors (B) leave the coordinator blind. Hiding errors as success (C) corrupts results. Full termination (D) is disproportionate for one timeout.
Reference Topics
Error propagation in multi-agent systems → Building with the Claude API — Agents and tools, Chaining workflows
Structured tool results → Building with the Claude API — Sending tool results, Tool schemas
Q9 D2 Task 2.3
A synthesis agent requires 2–3 verification round-trips per task (+40% latency). Analysis shows 85% are simple fact-checks; 15% require deep investigation. What is the best way to reduce this overhead?
Why A: Least-privilege design — give the synthesis agent only the tool it needs for the 85% common case, keeping it efficient. Full web search (C) is over-provisioned. Batching (B) delays the entire pass. Proactive caching (D) is unpredictable and may miss what's actually needed.
Reference Topics
Tool selection & least-privilege design → Building with the Claude API — Fine grained tool calling, Agents and tools
MCP sampling & tool scoping → MCP Advanced Topics — Sampling
Scenario 5 — Claude Code for Continuous Integration
Q10 D3 Task 3.6
A CI pipeline running claude "Analyze this pull request" hangs indefinitely waiting for interactive input. What is the correct fix?
Why A: -p / --print is the documented non-interactive flag for Claude Code — it runs the prompt and exits without waiting for further input. CLAUDE_HEADLESS (B) is not a real env var. Redirecting stdin (C) causes an error. --batch (D) does not exist.
Reference Topics
Claude Code non-interactive / CI mode (-p flag) → Claude Code in Action — Github integration
Claude Code in CI/CD pipelines → Claude Code 101 — How Claude Code works
Q11 D4 Task 4.5
Your team runs two workflows: (1) a blocking pre-merge check that must complete within seconds, and (2) an overnight technical debt report that runs at 2 AM. A manager proposes switching both to the Message Batches API for 50% cost savings. How should you evaluate this?
Why A: The Message Batches API has a processing window of up to 24 hours — fundamentally unsuitable for blocking pre-merge checks that must complete in seconds. The overnight report has no latency requirement, making it perfect for batch to capture the 50% savings.
Reference Topics
Message Batches API & when to use it → Building with the Claude API — Agents and workflows section
Cost optimization strategies → Building with the Claude API — Prompt engineering, Structured data
Q12 D4 Task 4.6
A single-pass review of a 14-file PR produces inconsistent results: superficial comments on some files, missed bugs in others, and contradictory feedback across files. How should you restructure the review process?
Why A: Splitting directly addresses attention dilution — per-file passes ensure consistent depth for each file, and a separate integration pass catches cross-file issues. A larger context window (C) doesn't fix attention dilution. Three passes (D) adds cost without improving per-file depth.
Reference Topics
Context window management & attention → Building with the Claude API — Structured data, Structured data exercise
Chaining & parallelization workflows → Building with the Claude API — Chaining workflows, Parallelization workflows
